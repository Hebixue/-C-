# 门把手移植调试复盘

## 背景

本轮调试目标是把 `PE_Demo\PE2\03_Handle` 中已经验证过的门把手检测逻辑，移植到 `PE_Demo\PE2\02_LF_RF` 中，并和现有 CAN、RF、LF、继电器控制流程结合。

最终结论是：门把手检测本身可以正常移植，但把门把手触发和 LF 发射流程直接耦合后，会破坏原版 `02_LF_RF` 的 LF/RF 路线，表现为钥匙 LED 不闪烁、RSSI 不返回。因此最后回退到“原版 LF/RF 主流程 + CAN/继电器/协议层”的方案，暂时不接门把手。

## 第三次按下失效的问题

最早的问题现象是：

- 门把手第一次、第二次触发还能点亮 LED1。
- 第三次之后 LED1 不再点亮。
- 后续门把手看起来像是卡死。

当时移植后的代码只保留了初始化和简单事件清除：

```c
DH_Init();
DH_ClearEvent();
lf_ata5293_RFS();
checkStatus();
```

主循环里只是读取事件并清除：

```c
if (event != 0u)
{
    DH_ClearEvent();

    if (event & 0x01u)
    {
        LED1_ON;
        led1_tick = 0u;
    }
}
```

这个逻辑的问题是：只清了事件标志，但没有完整恢复 ATA5293 的门把手检测状态。

## 曾经尝试的修复

### 1. 补完整 DH_Rearm

根据早期分析，补了完整的重布防流程：

```c
static void DH_Rearm(void)
{
    lf_ata5293_RFS();
    DH_ClearEvent();
    lf_ata5293_GID();
    lf_ata5293_RUN();
    (void)checkStatus();
}
```

理论目标是：

```text
触发一次
→ 清事件
→ 清故障
→ GID
→ RUN
→ 回到检测状态
```

### 2. 尝试加入 IRQ 锁存和恢复时间

为避免 IRQ 电平保持期间反复读 `DHES`，曾加入：

- `s_handle_irq_latched`
- `s_dh_recovery_ms`
- `DH_RECOVERY_MS`

思路是：

```text
IRQ 触发
→ 只处理一次
→ 等 IRQ 释放
→ 再允许下一次触发
```

但实际测试中，这套逻辑反而导致一次都不稳定。原因是 `03_Handle` 的原始可用代码并没有这个锁存状态机，它使用的是更直接的高电平判断和 `DH_GetEvent()` 读取。

### 3. 完整回搬 03_Handle 的门把手逻辑

后来直接把 `03_Handle` 的门把手逻辑搬到 `02_LF_RF`：

- IRQ 配置恢复为内部下拉。
- IRQ 判断恢复为高电平有效。
- 初始化恢复为：

```c
DH_Init();
DH_ClearEvent();
lf_ata5293_RFS();
checkStatus();
```

- 主循环恢复为原版方式：

```c
if ((GPIO_DRV_ReadPins(IRQ_GPIO) & (1u << IRQ_PIN)) != 0u)
{
    uint8_t event = DH_GetEvent();
    if (event != 0u)
    {
        DH_ClearEvent();

        if ((event & 0x01u) != 0u)
        {
            LED1_ON;
            led1_tick = 0u;
        }
    }
}
```

这个版本门把手恢复正常，第三次按下失效的问题消失。

## LED 时间异常

移植后发现 LED 点亮时间比 `03_Handle` 长。

原因是两个工程的 LF 定时器周期不同：

- `03_Handle`：`LF_TIMER` 周期是 `250000us`
- `02_LF_RF`：当时是 `1000000us`

原版 `03_Handle` 使用：

```c
#define LED_ON_TICKS 4u
```

在 `03_Handle` 中是：

```text
250ms * 4 = 1s
```

但在 `02_LF_RF` 中变成：

```text
1000ms * 4 = 4s
```

后来曾改成基于 `TIMER_GetMillis()` 的 1000ms 定时，避免依赖 LF 定时器周期。

## 接入 CAN 后的情况

门把手触发后，加入了 `0x301` 唤醒状态报文：

```c
CAN_App_SendSysState(PKES_SYS_LF_WAKEUP,
                     PKES_STATUS_LF_WAKEUP_OK,
                     PKES_REGION_UNKNOWN,
                     active_antenna,
                     PKES_LOCK_KEEP,
                     PKES_TRIGGER_HANDLE,
                     0u);
```

只加 CAN 报文、不发 LF 时，门把手仍然正常。

这说明 CAN 队列和 CAN 报文发送本身不是门把手卡死的根因。

## 接入 LF 后出现的新 bug

随后把门把手触发接到 LF：

```c
PKES_Core_HandleLFEvent();
```

该函数内部实际调用：

```c
SendLFWakeUp();
```

此后出现了新的严重问题：

- 门把手触发后，钥匙 LED 不闪烁。
- RSSI 没有正常返回。
- CAN 上看到的 `0x302` 可能只有 `0x84`，后面全 0。

后面确认 `0x84 00 00 00 ...` 不是实际 RSSI，而是之前主动遥控分支里伪造过一帧空 RSSI 报文，后来已删除。

## 为什么钥匙不闪

复盘后认为主要问题是：门把手触发 LF 后，为了恢复门把手检测，曾在每次 `SendLFWakeUp()` 后立刻调用 `DoorHandle_RestoreDhMode()`。

而这个函数里做了 ATA5293 硬复位：

```c
GPIO_DRV_WritePin(NRES_GPIO, NRES_PIN, 0);
DelayLoop(...);
GPIO_DRV_WritePin(NRES_GPIO, NRES_PIN, 1);
DelayLoop(...);
DH_Init();
DH_ClearEvent();
lf_ata5293_RFS();
checkStatus();
```

这很可能打断了 LF 发射流程。

原版 `02_LF_RF` 的逻辑是：

```c
SendLFWakeUp();
ant_poll_idx = (ant_poll_idx + 1u) % ANTENNA_COUNT;
active_antenna = antenna_list[ant_poll_idx];
```

原版发 LF 后不会立刻复位 ATA5293。  
如果 `SendLFWakeUp()` 返回时 LF 发送链路尚未完全稳定或仍需保持 TX/驱动状态，那么立刻复位就会导致钥匙收不到 LF，所以钥匙 LED 不闪。

## 最终回退方案

最后采用的方案是：

- 不再在当前阶段接门把手。
- 直接恢复 `02_LF_RF_原版` 的 LF/RF 主流程。
- 保留我们自己的：
  - CAN 非阻塞发送
  - 继电器控制
  - `0x301` 系统状态帧
  - `0x302` 钥匙信息与 RSSI 帧
  - `0x303` 测距与区域帧

当前主流程变为：

```text
定时 LF 轮询
→ SendLFWakeUp()
→ 切换 active_antenna
→ RF 收 5 字节
→ CRC 校验
→ PKES_Core_ProcessRFData()
→ 发送 CAN 报文 / 控制继电器
```

这个方案验证后，LF/RF 路线恢复正常。

## 后续建议

后面如果还要重新接门把手，不建议直接在门把手事件里调用 `SendLFWakeUp()` 并立即恢复 DH。

更稳的方向是：

1. 先保持 `02_LF_RF_原版` 的 LF/RF 路线完全不动。
2. 门把手只作为“启动一次原版 LF 轮询流程”的触发源。
3. 不在每次 LF 后复位 ATA5293。
4. 如果必须恢复 DH 检测，应等整轮 LF/RF 完成后再恢复。
5. 接入前要用示波器或逻辑分析仪确认 LF 输出确实存在，钥匙 LED 是否闪烁。

关键经验是：门把手检测和 LF 发射共用 ATA5293，不能把它们当成两个完全独立模块。初始化能跑通不代表状态机能共存，真正危险的是事件处理后的模式切换和硬复位时机。
