# CAN 报文重复发送临时关闭说明

## 背景

当前工程的 CAN 应用层原本设计为：每入队一帧报文，重复发送 3 次，每次间隔 100ms。

原代码配置在：

```c
#define CAN_APP_REPEAT_COUNT 3u
#define CAN_APP_REPEAT_DELAY_MS 100u
```

这样做的目的是提高评委工具或调试工具捕获事件帧的概率。

## 暂时关闭的原因

现在系统进入 LF/RF 被动 RSSI 调试阶段，4 根天线会按顺序轮询：

```text
天线1 → 天线2 → 天线3 → 天线4
```

每次 LF 轮询会发送 `0x301 LF_WAKEUP`，如果钥匙返回 RSSI，还会发送：

```text
0x302 钥匙信息与 RSSI 帧
0x301 RF_RECEIVED 状态帧
```

如果每帧都重复 3 次，CAN 发送队列会出现明显滞后：

```text
实际已经轮询到天线4
CAN 上可能还在发送天线1或天线2的旧报文
```

这不是报文内容被篡改，而是发送队列中旧帧还没有发完。

同时，CAN 队列长度只有 8：

```c
#define CAN_APP_QUEUE_SIZE 8u
```

如果入队速度超过发送速度，队列满后新帧会被直接丢弃：

```c
if (s_queue_count >= CAN_APP_QUEUE_SIZE)
{
    return;
}
```

因此在 RSSI 调试阶段，重复发送会影响判断“当前看到的 0x302 是否对应当前天线”。

## 当前修改

将重复发送次数从 3 次改为 1 次：

```c
#define CAN_APP_REPEAT_COUNT 1u
```

保留原有 CAN 队列状态机，不改发送接口、不改报文格式。

## 影响

修改后，每一帧 CAN 报文只发送一次。

优点：

- 明显减少队列堆积。
- `0x302` RSSI 帧更接近真实接收时刻。
- 更容易判断 `Byte3 RSSI_Ant` 是否对应当前天线。
- 减少队列满导致丢帧的概率。

代价：

- 调试工具如果偶发漏收 CAN 帧，不会再有后续重复帧补偿。

## 后续建议

RSSI 和天线轮询调通前，建议保持单次发送。

等流程稳定后，如果还需要提高报文捕获率，可以考虑只对关键低频事件重复发送，例如：

- 开锁完成
- 闭锁完成
- 系统异常

而不要对高频 RSSI 帧重复发送。
